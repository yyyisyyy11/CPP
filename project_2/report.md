# Dot Product Performance Benchmark Report

## 1. 实验概述

### 实验目标
对比 C 语言和 Java 在计算向量点积时的性能差异，覆盖多种数据类型和向量规模。

### 实验环境
- **平台**：macOS (Apple Silicon)
- **C 编译器**：gcc（两种配置：无优化 `-O0`，最高优化 `-O3`）
- **Java**：JDK（默认 JIT 编译）
- **随机种子**：42（固定，确保可复现）

### 实验设计
- **数据类型**：float (4B), double (8B), int (4B), short (2B), signed char / byte (1B)
- **向量规模 n**：128, 1000, 10⁴, 10⁵, 10⁶, 10⁷, 10⁸
- **计时方式**：C 使用 `<time.h>`，Java 使用 `System.nanoTime()`

## 3. 分析

### 3.1 C-O0 vs C-O3：编译器优化的效果

`gcc -O3` 在编译时对代码施加了一系列静态优化，效果在不同数据类型上差异显著：

| 数据类型 | O3/O0 加速比 | 分析 |
|----------|-------------|------|
| **int** | **8.41x** | 整数运算最受益：O3 可将 int 乘加循环充分向量化（NEON 128-bit 寄存器一次处理 4 个 int），且无浮点精度约束 |
| short | 3.50x | 向量化收益更高（128/16=8 个/次），但 short→int 的类型提升增加了额外指令 |
| signed char | 3.23x | 理论并行度最高（128/8=16 个/次），但类型提升开销更大 |
| float | 2.04x | 浮点向量化受限于 IEEE 754 精度要求，编译器不能随意重排加法顺序 |
| **double** | **1.72x** | 128-bit 寄存器只能装 2 个 double，向量化收益最小 |

**关键发现**：O3 对整数类型的加速远超浮点类型，这符合 SIMD 向量化的理论预期——元素越小并行度越高，且整数运算没有浮点精度约束。

### 3.2 Java 的 JIT（Just-In-Time）编译现象

#### 3.2.1 现象观察

Java 的运行时间出现了一个**反直觉**的现象：

| 数据类型 | n=10⁵ | n=10⁶ | 变化 |
|----------|--------|--------|------|
| float | 1.0985 ms | 1.0827 ms | n 增大 10 倍，时间反而**下降** |
| double | 1.1158 ms | 1.0584 ms | 同上 |
| int | 1.2583 ms | 0.3328 ms | 时间暴降 **73%** |
| short | 1.1252 ms | 0.3450 ms | 时间暴降 **69%** |
| byte | 1.1281 ms | 0.3578 ms | 时间暴降 **68%** |

此外，所有类型在 n=10⁵ 时耗时几乎相同（~1.1ms），与数据类型大小无关——暗示此阶段的耗时由某种固定开销主导，而非真正的计算。

**推测**：JVM 的 JIT 编译器在 n=10⁵ 到 10⁶ 之间完成了从解释执行到编译执行的切换。之前的数据点反映的是字节码解释器的性能，之后反映的是编译后的本机机器码性能。

#### 3.2.2 实验验证

**实验 1：固定 n，重复运行观察速度变化**

设计控制实验 `JitWarmup.java`：固定 `n=10,000`，重复调用 `dotFloat` 30 次。

```
Round     Time (ms)
─────     ─────────
  1        0.1140    ← 解释执行
  2        0.1132
  ...
  7        0.1100
  8        0.0235    ← 速度突变！（JIT 编译完成）
  9        0.0229
 ...       ~0.023    ← 此后稳定
```

- Round 1–7：平均 ~0.112 ms（解释执行）
- Round 8–30：平均 ~0.023 ms（编译后执行）
- **加速比**：0.112 / 0.023 ≈ **4.9x**

n 始终为 10,000，但速度在第 7→8 轮突然提升了近 5 倍——证明速度跳变纯粹是 JIT 编译的效果。

**实验 2：PrintCompilation 直接观察编译事件**

使用 JVM 诊断参数运行：

```bash
java -XX:+PrintCompilation Dotproduct.java 2>&1 | grep "Dotproduct::dot"
```

输出：

```
时间戳   编号   标记  层级  方法                             字节码
──────  ─────  ───  ───  ──────────────────────────────   ─────────
1325    2016    %    3    Dotproduct::dotFloat   @ 4      (28 bytes)
1325    2017         3    Dotproduct::dotFloat             (28 bytes)
3286    2044    %    4    Dotproduct::dotDouble  @ 5      (32 bytes)
3316    2046         4    Dotproduct::dotDouble            (32 bytes)
7000    2109    %    4    Dotproduct::dotInt     @ 5      (34 bytes)
7008    2112         4    Dotproduct::dotInt               (34 bytes)
8821    2125    %    4    Dotproduct::dotShort   @ 5      (34 bytes)
8835    2128         4    Dotproduct::dotShort             (34 bytes)
10583   2136    %    4    Dotproduct::dotByte    @ 5      (34 bytes)
10595   2143         4    Dotproduct::dotByte              (34 bytes)
```

字段含义：`%` = OSR编译（循环中热替换），层级 3 = C1 快速编译，层级 4 = C2 最高优化，`@ N` = OSR 入口点。

每个方法成对出现：OSR 版（替换当前循环）+ 标准版（供后续调用使用）。所有方法最终都被提升到层级 4（C2 编译器），编译时间戳与速度跳变完全吻合。

#### 3.2.3 JIT 的本质机制

JVM 采用**分层编译（Tiered Compilation）**策略：

```
阶段 0:   解释执行      → 逐条翻译字节码，最慢
阶段 1-3: C1 编译执行   → 快速编译，中等优化
阶段 4:   C2 编译执行   → 深度优化，生成高性能机器码
```

**触发条件**：基于代码执行次数，而非挂钟时间。循环体累计迭代 ~10,000 次后触发 OSR。本 benchmark 中按 n=128→1000→10000 顺序运行，累计 11,128 次刚好超过阈值。

**OSR（On-Stack Replacement）**：JIT 不需要等方法返回才能切换。当回边计数达到阈值时，JVM 在循环头处将解释执行的栈帧原地替换为编译后的机器码，后续迭代直接以本机代码运行。

**C2 编译器可能应用的优化**（`PrintCompilation` 证明 C2 被调用，但具体优化需查看机器码才能确认）：

| 优化技术 | 说明 |
|----------|------|
| 边界检查消除 | 识别出循环条件已保证安全，去掉数组越界检查 |
| SIMD 向量化 | 利用 NEON 128-bit 寄存器一次处理多个元素 |
| 循环展开 | 减少循环控制指令的占比 |
| 寄存器分配优化 | 将累加变量 `sum` 锁定在寄存器中 |
| 方法内联 | 将 dot 函数体嵌入调用处 |

### 3.3 数据类型间的差异

在 n=10⁸（JIT 已充分生效）时，不同数据类型的性能特征呈现清晰规律：

**整数类型 vs 浮点类型（C-O3）**

| 类型 | C-O3 耗时 (ms) | 相对 int 的倍数 |
|------|----------------|----------------|
| int | 13.81 | 1.0x |
| short | 32.28 | 2.3x |
| signed char | 34.41 | 2.5x |
| float | 192.00 | 13.9x |
| double | 367.65 | 26.6x |

整数运算大幅快于浮点运算。虽然 short/char 的理论 SIMD 并行度更高（8 个/16 个 vs int 的 4 个），但实际性能反而不如 int，原因是 C 语言的**整数提升（integer promotion）规则**：short/char 在参与运算前必须先被提升为 int，增加了额外的类型转换指令。

**浮点类型 double 为什么最慢？**

double（8字节）在 n=10⁸ 时 C-O3 耗时 367.65ms，是 float（192.00ms）的 1.9 倍。原因：
1. **SIMD 并行度减半**：128-bit 寄存器只能装 2 个 double vs 4 个 float
2. **内存带宽翻倍**：加载 10⁸ 个 double 需要 1.6GB 数据 vs float 的 0.8GB，受限于内存带宽

**Java 在浮点和整数上的不同表现**

在 n=10⁸ 时：
- **float/double**：Java 大幅快于 C-O3（2.0x / 3.4x），可能是 JIT 的运行时优化更激进
- **int**：C-O3 反而快于 Java（2.32x），说明 gcc 的静态整数向量化优化更成熟
- **short/byte**：Java 与 C-O3 基本持平

### 3.4 C-O0 vs Java 的对比意义

在大 n 时，Java **全面快于** C-O0（3.5x–5.8x）。这说明：

- **JIT 编译后的 Java 并不"慢"**——它慢的只是 warmup 阶段
- 未优化的 C 代码（-O0）性能很差，甚至不如带 JIT 的 Java
- 这提示我们：语言选择本身不是性能瓶颈，**编译器优化才是关键**

---

## 4. 总结

| 特性 | C (`gcc -O3`) | Java (JIT C2) |
|------|---------------|---------------|
| 优化时机 | 编译时（静态） | 运行时（动态） |
| 首次运行 | 从第一次就快 | 需要 warmup |
| SIMD | 编译器自动向量化 | C2 自动向量化 |
| Profile-Guided | 需手动 PGO | 自动 profiling |
| 边界检查 | 无（程序员负责） | 自动消除 |

**核心结论**：

1. **C 并不总是比 Java 快**。在大规模浮点计算中，JIT 编译后的 Java 甚至超过 C-O3
2. **编译器优化（-O3）的效果远超语言选择**。C-O0 vs C-O3 的差距可达 8.4x
3. **JIT 的 warmup 是 Java 小 n 下慢的主因**，而非语言本身的效率问题
4. **数据类型选择影响显著**：整数远快于浮点，小数据类型受制于类型提升
